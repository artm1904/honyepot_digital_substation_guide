### Шаг 1: Установка T-Pot в KVM

    Скачайте ISO образ  Ubuntu Server 24.04 LTS 

    Запустите Virt-Manager (из меню приложений или командой virt-manager).

    Нажмите "Создать новую виртуальную машину".

    Выберите ISO образ.

    Ресурсы: Выделите минимум 8 ГБ RAM и 70 ГБ диска.

    Сеть (Network Selection):
    На последнем шаге мастера (где имя ВМ) разверните вкладку "Network selection" (Выбор сети).

        Выберите Network source: Virtual network 'host-bridge': Bridge network.

        Выберите Bridge device... и впишите br0.

    Завершите создание и начните установку.

Важные нюансы для T-Pot

    Promiscuous Mode: T-Pot может потребовать перевода интерфейса ВМ в неразборчивый режим, но при использовании моста KVM обычно справляется сам.

    SSH доступ: T-Pot меняет порт SSH (обычно на 64295), чтобы освободить порт 22 для honeypot.

    Firewall на хосте :
    Поскольку мы используем Bridge, трафик идет сквозь ваш физический интерфейс сразу в ВМ (на уровне L2).

        iptables / nftables на самом Ubuntu Server 24.04 LTS не должны блокировать трафик, проходящий через мост (FORWARD chain).

        Вам, возможно, потребуется разрешить трафик через мост. Если у вас чистый Ubuntu без фаервола, всё заработает само. Если есть фаервол, добавьте правило разрешать всё на интерфейсе br0.

    Чтобы убедиться, что трафик ходит через мост, проверьте настройки ядра:

    ```bash
    sysctl net.bridge.bridge-nf-call-iptables
    ```

    Если там 1, то iptables хоста фильтруют трафик моста. Если будут проблемы с подключением к T-Pot извне, попробуйте установить это значение в 0 (или настройте правила iptables корректно).

Теперь ваш T-Pot будет виден в локальной сети как отдельное устройство со своим IP-адресом, полученным от вашего роутера.

Для проверки базовой работоспособности T-Pot, необходимо зайти в веб-интерфейс по адресу https://<IP_ADRESS_TPOT_VM>:64297 с паролем для веб-интерфейса, заданным при установке.

В случае если веб-интерфейс не открывается, необходимо проверить работу веб-сервера 

```bash
sudo docker ps -a

sudo docker logs container_name -f
```

Once the T-Pot Installer successfully finishes, the system needs to be rebooted (sudo reboot). Once rebooted you can log into the system using the user you setup during the installation of the system. Logins are according to the User Types:

    user: [<OS_USERNAME>]
    pass: [password]

You can login via SSH to access the command line: ssh -l <OS_USERNAME> -p 64295 <your.ip>:

    user: [<OS_USERNAME>]
    pass: [password, ssh key recommended]

    user: [<WEB_USER>]
    pass: [password]

Distributed Deployment
Planning and Certificates

The distributed deployment involves planning as T-Pot Init will only create a self-signed certificate for the IP of the Hive host which usually is suitable for simple setups. Since logstash will check for a valid certificate upon connection, a distributed setup involving Hive to be reachable on multiple IPs (i.e. RFC 1918 and public NAT IP) and maybe even a domain name will result in a connection error where the certificate cannot be validated as such a setup needs a certificate with a common name and SANs (Subject Alternative Name).



Теперь настроим архитектуру Distributed Deployment. В этой схеме одна виртуальная машина становится Hive (Сервер/Улей), который собирает логи и рисует графики, а остальные становятся Sensor (Сенсорами), которые просто ловят атаки и пересылают данные на Hive.

Важно: Установщик T-Pot (deploy.sh) работает по принципу "Push". Вы настраиваете всё на Hive, и он сам подключается к будущему Сенсору и переделывает его конфигурацию.

Предположим для инструкции:

    VM1 (Hive/Сервер): IP 192.168.1.200 (У вас будет свой, посмотрите через ip a)

    VM2 (Sensor/Сенсор): IP 192.168.1.201


ВМ должны быть установлены как Hive или Sensor


Этап 1: Подготовка сертификатов на Hive

Logstash (сборщик логов) очень строг к SSL-сертификатам. Если IP-адрес в сертификате не совпадает с реальным IP, логи приходить не будут. Стандартный сертификат создается для локального интерфейса, поэтому его нужно пересоздать с вашим реальным IP моста.

Выполняем на VM1 (Hive):

    Зайдите по SSH или через консоль в Hive.

    Остановите службы T-Pot:
    code Bash

    sudo systemctl stop tpot

    Сгенерируйте новый сертификат.
    ВНИМАНИЕ: В команде ниже замените IP:192.168.1.200 на реальный IP вашего Hive. Если у вас есть доменное имя, замените DNS:my.domain, если нет — просто удалите часть с DNS.
    code Bash

    sudo openssl req \
        -nodes \
        -x509 \
        -sha512 \
        -newkey rsa:8192 \
        -keyout "$HOME/tpotce/data/nginx/cert/nginx.key" \
        -out "$HOME/tpotce/data/nginx/cert/nginx.crt" \
        -days 3650 \
        -subj '/C=US/ST=Lab/O=TPot/CN=HiveRoot' \
        -addext "subjectAltName = IP:192.168.1.200"

    (Если у Hive несколько IP, например, внутренний и внешний, перечислите их через запятую: IP:192.168.1.200, IP:10.0.0.5).

    Настройте права доступа (это критично, иначе nginx не прочитает файлы):
    code Bash

    sudo chmod 774 $HOME/tpotce/data/nginx/cert/*
    sudo chown tpot:tpot $HOME/tpotce/data/nginx/cert/*

    Запустите T-Pot обратно:
    code Bash

    sudo systemctl start tpot




    Этап 2: Настройка SSH доступа (с Hive на Sensor)

Скрипт установки использует Ansible, которому нужно заходить с Hive на Sensor без пароля.

Выполняем на VM1 (Hive):

    Сгенерируйте SSH ключ (нажимайте Enter на все вопросы, парольную фразу оставляем пустой):
    code Bash

    ssh-keygen

    Скопируйте ключ на VM2 (Sensor).
    Замените user на имя вашего пользователя в T-Pot (обычно то, которое вы создавали при установке Debian/T-Pot).
    Замените 192.168.1.201 на IP вашего Сенсора.
    Порт 64295 — это стандартный SSH порт T-Pot.
    code Bash

    ssh-copy-id -p 64295 user@192.168.1.201

    (Вас попросят ввести пароль пользователя VM2 один раз).

    Обязательно проверьте, что заходит без пароля:
    code Bash

    ssh -p 64295 user@192.168.1.201

    Если зашли и не спросило пароль — отлично. Напишите exit, чтобы вернуться на Hive.

Этап 3: Запуск скрипта развертывания

Теперь самое интересное. Скрипт превратит вторую ВМ в подчиненный сенсор.

Выполняем на VM1 (Hive):

    Перейдите в папку с T-Pot:
    code Bash

    cd ~/tpotce

    Запустите деплой:
    code Bash

    ./deploy.sh

    Мастер задаст вопросы:

        Select installation type: Выберите Hive (так как мы находимся на нем и хотим добавить сенсоры к нему). Иногда меню может отличаться в версиях, ищите пункт "Deploy to Sensor" или подобное, но обычно запускают на Hive.

        Если скрипт спросит SSH пользователя для сенсора — введите имя пользователя (как в шаге 2).

        Если спросит IP сенсора — введите IP VM2 (192.168.1.201).

Скрипт автоматически:

    Подключится к VM2.

    Установит необходимые конфиги.

    Настроит Filebeat/Logstash на отправку логов на IP вашего Hive (используя сертификат, который мы сделали в Этапе 1).

    Перезагрузит VM2.

Этап 4: Проверка

    Подождите, пока VM2 (Sensor) перезагрузится.

    Зайдите в веб-интерфейс Hive: https://192.168.1.200:64297.

    Откройте Kibana -> Dashboard -> T-Pot.

    В фильтрах или в поле "Honeypot Sensor" вы должны начать видеть не только имя вашего Hive, но и имя вашего Sensor.

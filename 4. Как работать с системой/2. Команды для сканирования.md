
- сканирование через nmap (долгое)
```
sudo nmap -sV IP
```
- `-sV`, Nmap проходит через несколько этапов:
- **Сначала** он сканирует **1000 самых популярных портов**, чтобы найти открытые.
- **Затем** для каждого открытого порта он пытается «снять баннер» (приветствие сервиса).
- **Если баннера нет**, он отправляет специальные запросы и сравнивает ответы со своей огромной базой отпечатков (fingerprints).


- сканирование через nmap (быстрое)

```
sudo nmap -n -T4 -F -sV --version-light --open 192.168.1.0/24
```

- проверить конкретный порт
``` 
sudo nmap -p 102 -Pn --open 192.168.1.201 
```

- Тут уже больше банеров, но дольше
```
sudo nmap -sV -F -O -T4 IP
```
- `-F` (Fast): сканирует только 100 самых популярных портов вместо 1000.
- `-T4`: ускоряет процесс за счет уменьшения задержек между пакетами (подходит для стабильных сетей).
- `-O`: включает определение операционной системы. Nmap анализирует, как стек TCP/IP цели реагирует на разные пакеты, и угадывает, что там стоит (Linux, Windows, Cisco и т.д.).




## Пример сканирования

Результат сканирования подтверждает: **порт 102 открыт**, но Nmap немного «запутался» в том, что именно там работает.
### Почему так произошло?
1. **Ложное имя сервиса (`ms-wbt-server`):** Nmap увидел порт 102 и подставил имя из своей базы данных (иногда этот порт ассоциируется с RDP/Terminal Services). На самом деле у вас там **ISO-TSAP** (протокол, на котором базируются S7 и IEC 61850). Не обращайте внимания на название `ms-wbt-server`.
2. **Магия в отпечатке (Fingerprint):** Посмотрите на начало вашего сервис-фингера: `\x03\0\0\x16`.
    - `\x03` — это версия протокола TPKT.
    - `\0\0\x16` — длина пакета (22 байта). Это классический заголовок для промышленных протоколов. То, что скрипт `s7-info` не сработал, означает, что ваш хост (`192.168.1.201`) эмулирует **IEC 61850**, а не Siemens S7. Скрипт `s7-info` ожидает специфический ответ от ПЛК Siemens, которого honeypot не дает.

sudo nmap -p 102 -sV --script s7-info 192.168.1.201

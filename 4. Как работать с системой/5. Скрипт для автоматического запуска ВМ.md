📖 Руководство пользователя: Система управления и мониторинга виртуальных инфраструктур (check_vms.py)
1. Назначение программы (Для чего нужен скрипт?)

Скрипт check_vms.py — это мощный инструмент системного администратора и специалиста по ИБ, созданный для автоматизированного управления, настройки и глубокого мониторинга сложных виртуальных инфраструктур (киберполигонов, сетей ханипотов, стендов АСУ ТП), работающих под гипервизором QEMU/KVM.

Главная проблема, которую решает скрипт: управление изолированными машинами без доступа к сети. Скрипту не нужен SSH или работающий DHCP. Он "общается" с виртуальными машинами напрямую через аппаратную шину гипервизора.
2. Основные возможности (Что он делает?)

    Умный автозапуск: Скрипт проверяет состояние машин. Если они выключены — запускает их и ждет полной загрузки операционной системы.

    «Out-of-band» конфигурация сети: Если у ВМ нет нужного IP-адреса, скрипт мгновенно назначает его на нужный интерфейс прямо "извне".

    Многопоточный опрос (Турбо-режим): Одновременно (в 20 потоков) сканирует десятки узлов: пингует их и проверяет доступность нужных TCP-портов.

    Глубокий сбор телеметрии: Запрашивает изнутри каждой ВМ реальную загрузку CPU, оперативной памяти (RAM), жесткого диска (Disk) и время непрерывной работы (Uptime).

    Анализ Docker-контейнеров: Считает количество работающих контейнеров и отдельно выявляет упавшие (Exited/Dead). Также замеряет вес директории с логами.

    Радар кибератак (Live Attack Indicator): В реальном времени анализирует сетевые сокеты и выводит таблицу с внешними IP-адресами злоумышленников, которые прямо сейчас атакуют ваши сервисы.

3. Системные требования

На хосте (откуда запускается скрипт):

    ОС Linux (Arch, Debian, Ubuntu и т.д.).

    Python версии 3.6 и выше.

    Установленные библиотеки: pip install pyyaml.

    Гипервизор QEMU/KVM (утилита virsh).

    Права root (скрипт запускается через sudo, чтобы иметь доступ к virsh).

Внутри виртуальных машин (Гостевые ОС):

    ОС Linux (Ubuntu, Debian, Arch и т.д.).

    Критически важно: Должен быть установлен пакет qemu-guest-agent и запущен как системная служба:
    code Bash

    sudo apt install qemu-guest-agent
    sudo systemctl enable --now qemu-guest-agent

4. Настройка конфигурационного файла (config.yaml)

Файл config.yaml должен лежать в той же папке, что и скрипт. Он написан в формате YAML (важно соблюдать отступы пробелами!).

В нем описывается вся ваша инфраструктура: какие машины существуют, какие у них IP-адреса и какие порты на них нужно проверять.
Структура config.yaml:
code Yaml

vms:
  # Пример 1: Обычная виртуальная машина (Скрипт настроит ей сеть)
  - name: "Ubuntu-Main"         # ТОЧНОЕ имя ВМ в гипервизоре (команда: virsh list)
    interface: "enp1s0"         # Имя сетевого интерфейса внутри ВМ
    ip: "192.168.1.100/24"      # IP-адрес с маской подсети
    gateway: "192.168.1.1"      # Шлюз по умолчанию
    services:                   # Список портов для проверки
      - { port: 22, name: "SSH Server" }
      - { port: 80, name: "Web API" }

  # Пример 2: Docker-контейнер (сеть Macvlan), работающий внутри Ubuntu-Main
  - name: "Honeypot-DB"         # Произвольное красивое имя для дашборда
    ip: "192.168.1.205"         # IP-адрес контейнера (скрипт будет его пинговать)
    check_only: true            # ФЛАГ ПРОПУСКА НАСТРОЙКИ (Критически важно!)
    services:
      - { port: 5432, name: "PostgreSQL" }

❗️ Что такое check_only: true?

Поскольку в вашей системе ханипоты запущены в Docker (через macvlan), они выглядят как отдельные компьютеры с отдельными IP-адресами, но физически живут внутри одной ВМ.

    Если не указать этот флаг, скрипт попытается "залезть" в контейнер через гипервизор и сломает его.

    Флаг check_only: true говорит скрипту: "Не пытайся настраивать тут сеть или вытаскивать CPU. Это просто виртуальный IP. Просто попингуй его и проверь порты".

5. Инструкция по запуску

Откройте терминал в папке со скриптом и выполните команду:
code Bash

sudo python check_vms.py

(Используется sudo, так как управление гипервизором требует привилегий администратора).

Скрипт работает полностью в автоматическом режиме. Его алгоритм:

    Этап 0: Читает конфиг. Находит выключенные ВМ и посылает команду на их включение. Ждет, пока внутри не запустится гостевой агент (ОС загрузилась).

    Этап 1: Заходит в каждую ВМ. Проверяет, правильный ли там IP. Если нет — вежливо прописывает IP-адрес и шлюз.

    Ожидание: Если ВМ включались с нуля, скрипт покажет красивый прогресс-бар и подождет 2 минуты, чтобы тяжелые Docker-контейнеры успели стартовать. Если ВМ уже работали — этот шаг пропускается.

    Этап 2 (Сканирование): Скрипт мгновенно опрашивает все узлы, собирает метрики, проверяет порты и рисует иерархическую цветную таблицу.

    Этап 3 (Радар атак): Скрипт собирает список всех "чужих" сетевых подключений и выводит их в красную таблицу.

6. Расшифровка интерфейса (Как читать Дашборд)

Дашборд состоит из строк, объединенных в "деревья" (где дочерние элементы — это Docker-контейнеры).
Цветовая индикация (Светофор)

    🟢 Зеленый: Метрика в норме (< 50% нагрузки) / Порт открыт / Узел доступен.

    🟡 Желтый: Повышенная нагрузка (50% - 80%). Стоит обратить внимание.

    🔴 Красный: Критическая перегрузка (> 80%) / Порт недоступен / Узел мертв.

Значение колонок:

    УЗЕЛ / КОНТЕЙНЕР: Имя из конфига. ВМ выделены жирным шрифтом, контейнеры сдвинуты вправо (ветви ├──).

    IP АДРЕС: Сетевой адрес узла.

    CPU: Процент загрузки процессора в реальном времени.

    RAM: Процент занятой оперативной памяти.

    DISK: Процент заполненности корневого раздела диска (/).

    UPTIME: Время работы ВМ с момента последнего включения (например, 2d 14h — 2 дня, 14 часов).

    ЛОГИ: Размер директории с логами T-Pot (помогает отслеживать переполнение диска).

    DOCKER: Количество работающих контейнеров.

        Внимание: Если вы видите статус вида 21 (⚠2) красным цветом, это означает, что 21 контейнер работает, но 2 контейнера упали с ошибкой.

    PING: Статус доступности по сети (UP / DOWN).

    СТАТУС СЕРВИСОВ: Список проверенных портов. Зеленая точка ● означает, что сервис работает, красная ○ — сервис "лежит".

Как читать Радар Активных Атак (Этап 3)

Если в вашей сети (или в интернете) кто-то сканирует ваши ВМ или подбирает пароли, скрипт выведет красную таблицу:

    ЦЕЛЬ: Какой именно ваш ханипот атакуют.

    ПОРТ: На какой порт идет атака (например, SSH или HTTP).

    АТАКУЮЩИЙ (IP): IP-адрес злоумышленника.

    КОЛ-ВО СОЕДИНЕНИЙ: Сколько одновременных потоков открыл хакер. Если их больше 5, скрипт подпишет: (DDoS / Bruteforce?).

7. Возможные проблемы и их решение

1. Скрипт пишет: [ВМ] ❌ Превышено время ожидания. Агент не ответил.

    Причина: Внутри этой виртуальной машины не установлен (или завис) пакет qemu-guest-agent.

    Решение: Зайдите в ВМ вручную (через консоль virt-manager) и выполните sudo systemctl restart qemu-guest-agent.

2. Скрипт падает с ошибкой yaml.parser.ParserError

    Причина: Вы нарушили структуру отступов в файле config.yaml. В языке YAML нельзя использовать табуляцию (Tab), только пробелы.

    Решение: Проверьте конфиг, убедитесь, что перед - name: стоит ровно 2 пробела, а перед ip: — 4 пробела.

3. Узел пингуется (UP), но все порты красные (DOWN)

    Причина: ВМ включилась, сеть работает, но сервис внутри ВМ (например, Docker или Nginx) не запустился, либо доступ к порту заблокирован фаерволом (iptables).

    Решение: Зайти на ВМ и проверить логи проблемного сервиса (systemctl status docker или docker logs <container>).

4. Все узлы DOWN (не пингуются) после запуска

    Причина: Неверно указано имя интерфейса (например, в конфиге написано enp1s0, а в Ubuntu он называется eth0), либо неверно указан шлюз gateway.

    Решение: Исправьте параметр interface: в config.yaml на правильный. Скрипт при следующем запуске автоматически перенастроит сеть.

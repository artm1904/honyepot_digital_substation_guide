Архитектура представляет собой изолированный центр мониторинга (HIVE) и высокореалистичный сегмент АСУ ТП (Sensor). Для обхода детектирования honeypot-ов злоумышленниками применена технология L2-мостов (macvlan), которая присваивает каждому эмулируемому контроллеру уникальный IP-адрес в целевой подсети. Внутри узлов-сенсоров настроена маршрутизация на основе политик (Traffic Control) для изоляции внутреннего легитимного трафика симулятора от внешнего (атакующего) трафика, а также внедрена гибридная логика: эмуляция протоколов IEC 61850 работает параллельно с LLM-движком, динамически генерирующим ответы для системных сервисов (SSH/Web) через внешнее API Ollama. Системы сетевого мониторинга (Suricata, p0f) прозрачно "слушают" хост и передают события в единый центр (ELK) без раскрытия своей позиции атакующему.

1. Глобальная архитектура (Взаимодействие ВМ)

У тебя работают две основные виртуальные машины:

    HIVE (Центральный узел / Мозг): Отвечает за сбор, хранение (Elasticsearch) и визуализацию (Kibana, Attack Map) данных. Не светит наружу уязвимыми портами.

    Sensor (Узел-приманка / Жертва): Эмулирует индустриальную сеть (подстанцию АСУ ТП) и уязвимые сервисы (SSH, Web). Принимает на себя удары и пересылает логи на HIVE.

    Опциональный внешний узел: Сервер с Ollama (IP 192.168.1.173), который используется сенсором для генерации ответов ИИ.

Поток данных (Data Flow):
Атакующий бьет по IP-адресам Сенсора -> Сенсор генерирует логи -> Logstash на Сенсоре забирает логи и по зашифрованному каналу (через custom_http_output.conf) отправляет их на порт 64305 ВМ HIVE -> Logstash на HIVE кладет их в Elasticsearch -> Администратор смотрит красивые графики через Kibana.
2. Сетевая архитектура ВМ HIVE

Здесь сеть построена максимально безопасно, чтобы защитить собранные данные от компрометации.

    Управление хостом: Нативный SSH-сервер ОС перенесен на нестандартный порт 64295, чтобы избежать конфликтов с ханипотами и отсеять тупые автосканеры.

    Изоляция контейнеров: Контейнеры ELK-стека (Elastic, Logstash, Kibana, Redis) общаются друг с другом по изолированной Docker bridge-сети nginx_local. Они не торчат наружу напрямую.

    Реверс-прокси (Nginx): Единственная точка входа для администратора. Nginx принимает подключения на портах 64294 и 64297 и маршрутизирует их на внутренние интерфейсы Kibana, Attack Map и Spiderfoot.

    Прием логов: Logstash "слушает" порт 64305 (проброшен на 127.0.0.1 и управляется Docker Proxy), ожидая входящие логи от твоего Сенсора.

3. Сетевая архитектура ВМ Sensor (Самое интересное)

Сетевая модель сенсора имитирует архитектуру реальной подстанции стандарта IEC 61850. Для обмана атакующих используется гибридная сеть из macvlan и изолированных bridge-сетей.
А. Внешняя сеть (Сцена для хакера) - scada_macvlan

    Как работает: Вместо того чтобы прятать ханипоты за NAT-ом докера, ты использовал драйвер macvlan, привязанный к физическому/виртуальному адаптеру хоста (enp1s0).

    Эффект: Контейнеры IED (ied1_xcbr, ied2_ptoc и др.), а также Симулятор и Клиент получают реальные IP-адреса из твоей подсети (192.168.1.202 - 192.168.1.208).

    С точки зрения хакера: Сканируя сеть 192.168.1.0/24, злоумышленник видит россыпь независимых физических устройств (реле защиты, контроллеры), а не один сервер с кучей открытых портов.

Б. Внутренняя сеть (Скрытая логика) - subnetwork1

    Как работает: Классическая Docker bridge сеть (10.0.0.0/24).

    Назначение: Используется для служебного общения контейнеров между собой в обход "глаз" атакующего. Симулятор (10.0.0.254) общается с IED узлами и генерирует легитимный фоновый трафик, чтобы сеть казалась живой.

В. Магия внутри контейнеров IED (Двойное дно)

Каждый ханипот IED (например, ied1_xcbr) является "контейнером-матрешкой". Взаимодействие настроено через скрипт entrypoint.sh и утилиту tc (Traffic Control):

    Два интерфейса: Контейнер имеет два IP (один 10.0.0.x внутренний, один 192.168.1.x внешний).

    Зеркалирование трафика (Port Mirroring):

        Протокол слушает внутренний интерфейс, но скрипт tc filter на лету перехватывает весь MMS/GOOSE трафик (порт 102) с внешнего macvlan интерфейса и пересылает его внутрь.

        Ответы от симулятора зеркалируются обратно наружу.

        Зачем это нужно? Это позволяет разделить логику симулятора и точки входа атакующего, создавая иллюзию сложного аппаратного комплекса.

    Сожительство с ИИ (Beelzebub): На этом же контейнере в фоновом режиме висит LLM-ханипот. Если хакер решит, что это Linux-устройство, и постучится по SSH (порт 22) или HTTP (порт 80), отвечать ему будет не индустриальный протокол, а Beelzebub.

Г. Интеграция с LLM (Внешний вызов)

Когда Beelzebub ловит активность хакера (например, попытку ввести команду в SSH-терминале), контейнер делает HTTP POST запрос на внешний сервер 192.168.1.173:11434 (Ollama), передает команду хакера и контекст, получает сгенерированный нейросетью ответ и выводит его хакеру на экран.

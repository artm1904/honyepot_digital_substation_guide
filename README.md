# Техническое описание и руководство по эксплуатации: Программно-аппаратный комплекс (ПАК) "Система анализа действия нарушителей",выполняющий функционал имитации инфраструктуры цифровой подстанции


## 1. Методология и принципы разработки
Данный проект посвящен созданию исследовательской среды (стенда) для анализа киберугроз в сетях промышленной автоматики, работающих по стандарту **IEC 61850**. При проектировании архитектуры учитывались следующие определяющие факторы:

- **Ролевая модель и междисциплинарный подход:** Реализация проекта осуществлялась силами одного специалиста, что потребовало совмещения ролей системного архитектора, сетевого инженера, DevOps-специалиста, аналитика и пинтест-специалиста. Ввиду того, что основной фокус исследования сосредоточен на прикладных аспектах промышленной кибербезопасности (IEC 61850), реализация сопутствующих инфраструктурных задач выполнена на уровне, достаточном для создания работающего прототипа (PoC), но оставляющем пространство для дальнейшей экспертной оптимизации в смежных областях.   Автор осознает потенциальные точки роста в данных областях, которые выходят за рамки текущей специализации, и рассматривает их как векторы для привлечения профильных компетенций в будущем. Приоритет был отдан функциональной части ханипотов и логике взаимодействия по стандарту IEC 61850, в то время как вспомогательные системы (DevOps, сетевой инжиниринг) реализованы в базовой конфигурации.

- **Адаптивное проектирование:** Архитектура стенда была скорректирована с учетом имеющихся аппаратных мощностей (ограничение RAM/CPU, использование ноутбука с одним сетевым интерфейсом).
    
- **Итеративный подход:** Ввиду временных ограничений, реализация велась поэтапно, от базовых функций к более сложным механизмам взаимодействия. 

- **AI-Assisted Development:** При проектировании компонентов и разработке специализированного ПО (ханипотов) активно применялись большие языковые модели (LLM) для оптимизации кода и решения задач отладки.
    
В последующих разделах приведено детальное описание процесса развертывания инфраструктуры «с нуля», включая конфигурационные файлы, описание архитектуры разработанных ханипотов и инструкции по контейнеризации компонентов.


## 2. Текущая архитектура и системные ограничения
На текущем этапе стенд реализован на базе **Arch Linux** с использованием стека **QEMU/KVM**. Несмотря на работоспособность системы, выявлен ряд архитектурных особенностей и ограничений, которые необходимо учитывать при эксплуатации и дальнейшем развитии:

### 2.1. Сетевая топология и изоляция

В текущей итерации все компоненты (IT-ханипоты, IED-ханипоты, локальная LLM и системы мониторинга) функционируют в едином широковещательном домене (L2-сегменте).

- **Проблема:** Отсутствие демилитаризованной зоны (DMZ) и фактического разделения трафика между точками присутствия злоумышленника и внутренними сервисными компонентами.
    
- **Причина:** Ограничения физических интерфейсов хоста и отсутствие выделенного сетевого оборудования.
    
- **Зависимость:** Система жестко привязана к адресации `192.168.1.0/24`, что ограничивает гибкость развертывания в сторонних сетевых средах.



## 3. Дорожная карта модернизации (Roadmap)

Переход от связки **Arch Linux + QEMU/KVM** на ноутбуке к более структурированной лабораторной среде — это логичный шаг для улучшения инфраструктуры стенда. Изменения необходимы для  глубокого моделирования атак на промышленную автоматику (особенно IEC 61850)
Для повышения реалистичности моделирования атак и улучшения масштабируемости стенда определены следующие векторы развития:

### 3.1. Переход на гипервизор 1-го типа (Bare-metal)

Внедрение решений класса **Proxmox VE**, **ESXi** или **XCP-ng** позволит:

- **Минимизировать оверхед:** Исключение прослойки ОС общего назначения высвободит ресурсы CPU и RAM для гостевых систем.
    
- **Обеспечить аппаратный Pass-through:** Прямой проброс сетевых карт в виртуальные машины для минимизации задержек при обработке трафика.
    
- **Улучшить оркестрацию:** Использование снапшотов и API для автоматического отката состояния ханипотов после инцидентов.
    

### 3.2. Внедрение Программно-определяемых сетей (SDN)

- Для глубокого исследования протоколов **GOOSE** и **Sampled Values (SV)**, работающих на уровне L2, необходимо использование **Open vSwitch (OVS)**. Для исследования кибербезопасности в энергетике (IEC 61850) SDN — это не просто «улучшение», а критически важный инструмент.
- В реальных подстанциях трафик GOOSE и SV (Sampled Values) работает на втором уровне (L2) модели OSI. Обычные виртуальные мосты (`linux-bridge`) часто ограничены в возможностях мониторинга и фильтрации такого трафика. Можно заменить стандартные мосты KVM на OVS. Это позволит гибко управлять тегированием VLAN и упростит интеграцию с системами анализа трафика.
### Что даст внедрение SDN (например, Open vSwitch):
- **Сегментация:** Формирование изолированных шин процесса (Process Bus) и шин станции (Station Bus).
    
- **Зеркалирование (SPAN/RSPAN):** Организация прозрачного сбора трафика для систем обнаружения вторжений (IDS), таких как Suricata или Zeek.
    
- **Эмуляция среды:** Программное внесение задержек и джиттера для проверки устойчивости ханипотов в условиях нестабильной сети.
    

### 3.3. Виртуализация сетевого уровня (Virtual Routing)

Интеграция виртуального роутера (на базе **VyOS** или **pfSense**) позволит:

- Абстрагировать стенд от физической сети, обеспечив корректную маршрутизацию между подсетями IT и OT (IED) сегментов. Позволит сделать стенд переносимым его обеспечит его установку в любую инфраструтуру. Именно **pfSense** будет создавать сети IT, OT, внутренние для функционирования систем стенда.
    
- Реализовать детальное логирование соединений на сетевом и транспортном уровнях еще до попадания трафика в ханипоты.
    

### 3.4. Оптимизация системы мониторинга

Несмотря на текущее использование стека ELK, рассматривается переход на более легковесные решения для экономии ресурсов:

- **Grafana Loki:** Для сбора логов с меньшими затратами на индексацию.
    
- **Prometheus:** Для мониторинга метрик производительности системы в реальном времени.
